# Three-Tab Transactions Plan Suite — Lead Review

**VERDICT: `ready-after-blockers`** — strong, build-ready suite blocked only by a handful of small, surgical contradictions an autonomous agent would trip on.

## Executive summary

This is a mature, well-calibrated plan suite for a 20-person Verein — not enterprise gold-plating. The accounting model is correct (Sphäre strictly from Kategorie, Sachspende Wertermittlung per §9 BewG, EÜR degradation handled), the parallelization design is genuinely conflict-free where it matters most (the C1∥C2∥C3 tab tier), TDD discipline and targeted-test hygiene are consistently enforced, and the cross-cutting UX is one coherent product across three tabs. The blockers are not structural — they are local contradictions and unpinned contracts that would stall or silently mislead an autonomous overnight agent: a self-contradicting ROADMAP (transactions.ts ownership, money-input build-vs-forbid), an unpinned kategorie-filter value contract that silently returns zero rows, a Phase-4 task body that calls a non-existent parameter on the wrong helper, an importer that ships a CHECK violation the plan claims is impossible, and — the most important UX gap — no specified "neu" entry point on any tab. Every blocker is a one-to-three-line edit. Land those plus the spec's Sachspende-Kategorie 4th-combination gap and start building.

## Per-axis verdicts

**Implementation quality — strong.** The data-layer foundation (Phase 1) was validated against real code: migration numbering, journal shape, additive→wipe→constrain ordering, and the NOT-NULL closure across all write paths all check out. `resolveKategorieByName(kind,name)` is sound and backed by a real unique index. The filter backbone (Phase 2) is correctly load-bearing, not overbuilt. The defects are local: wrong `allocateBusinessId` prefixes in test snippets, unannotated `const c = []` builders, and one genuinely dangerous unpinned contract (kategorie filter value = name-snapshot vs id, P2-04). Most are typecheck-catchable; one is not.

**Self-review quality / targeted-test discipline — strong.** Every logic task has a failing test first, exact run commands, and tightened "Expected: N passed" signals. The fast-lane/reset-lane split is real and well-applied; phase-boundary gates enumerate explicit file lists rather than bare `pnpm test`. Two real gaps: the A3 "dependency-free" claim is prose-asserted (mostly enforced by `pnpm check`, so low), and three Tier-C boundary steps use a `git ls-files` glob that silently no-ops to green under `--passWithNoTests` (X-SELF-01). Two A3 tests under-cover the behavior their own implementation steps specify (Backspace, tooltip body).

**UX & delight — world-class intent, one load-bearing gap.** This is the strongest axis conceptually: one sphere palette, one list scaffold, one entry-form shell, one detail shell, one Beleg viewer, disappearing zero-pills, year-named empty states, and a loud non-dismissible stale-year banner mean Julia gets one product, not three screens. The opinionated calls are right (Sphäre as a quiet left-rule not a badge; no fake Sammelbestätigungs deadline; Spenden omitting a constant-color rule is correct information design, not an inconsistency). **But the single most-used action — "create a new booking" — has no specified entry point on any tab (UX-01). That is a real blocker:** three parallel C-tracks would each invent their own button placement/label/mobile treatment because the shared scaffold has no `newHref`/`newLabel` slot. Secondary UX gaps worth fixing: the spec'd Belegprüfung→Ausgaben handoff confirmation is absent from Phase 4 (UX-03), and the no-matches-vs-no-rows empty state lacks a reset affordance (a mobile dead-end trap, UX-04).

**Completeness — strong traceability, a few unowned spec promises.** Every mandatory-Kategorie/Beleg/sphere-derivation/year-scope/detail/viewer requirement maps to an owning task, and the cross-phase data dependencies are explicitly amended into Phases 1/3. The real gaps: the spec's headline "EÜR/dashboard reflects new bookings" (§16 E2E) has no owning verification task (P12-01, medium) and the showcase corpus's paid-invoice→income link is left conditional ("if helpers are readily available") while Phase 5 tests hard-require it (P47-05, medium). The Sachspende-Kategorie derivation has a genuine 4th-combination hole (SPEC-01, high) that breaks the NOT-NULL constraint at runtime.

**Parallelization — unusually disciplined, undercut by ROADMAP self-contradictions.** The hardest risk (the tab tier) is genuinely conflict-free: Phase 3 front-loads every cross-tab component and threads all per-kind detail fields through `getTransactionDetail` so no C-track touches `transactions.ts`; each tab's KPI lives in its own file; spenden + the Zuwendungsbestätigung move are solely C3. No two C-tracks own the same file. The problems are concentrated in the ROADMAP's own consistency: `transactions.ts` co-owned by A1/A2 inside the same concurrent tier (RM-01, high), `money-input` listed as build-it AND forbidden (RM-02, high), BelegViewer "A or B" ambiguous ownership (RM-03), and several stale cross-references (the Bulk/SEPA move attributed to the wrong task, PAR-04).

**Implementation autonomy — good but leaky at the edges.** The per-phase protocol (self-review → adversarial subagents → iterate → task-by-task) supports unattended execution. The leaks are amendment-blocks that correctly decide something but don't propagate the decision into the step bodies the agent actually executes: Phase 4 notify-helper (P4-01, high), Phase 5 filename field (P5-01), Phase 6 double-audit emit (X-PRAG-05/P6-04), and several "decide at impl" / "pick one" placeholders that an autonomous agent cannot resolve (P4-02, P6-02). These are the recurring autonomy hazard across the suite: the fix is always "make the step body agree with the amendment."

**Pragmatism / overengineering — well-calibrated.** The fast-lane config and the typed filter registry are genuine complexity that earns its keep at this scale. No structural overbuild. The saved-views rename/delete is spec-mandated and cheap, not gold-plating. The one item to flag-not-build: before hand-rolling combobox keyboard nav on a dumb popover, check whether bits-ui v2 ships a `Combobox` primitive (X-PRAG-06). The bulk-SEPA inbox carve-out adds disproportionate operational risk for a Verein and should be sequenced out of the C1 worktree (X-PRAG-03).

## Blockers
*Must fix before autonomous execution — each would stall or silently mislead an agent.*

- **RM-01 · ROADMAP · Tier A / rule line 82** — `transactions.ts` is edited by both A1 (create fns/sentinel) and A2 (listXPage) but they sit in the same concurrent tier; neither owns the file in its `owns:` list, and the footnote contradicts the tier header. An orchestrator assigns A1/A2 to concurrent worktrees and collides. **Fix:** add `transactions.ts` to A1's `owns:` as sole owner; sequence A2's listXPage after A1's commit; delete the contradicting footnote.
- **RM-02 · ROADMAP · A3 lines 50/52 vs 83** — `money-input` is listed as an A3 primitive to build AND explicitly forbidden ("ui/money already exists; do NOT create a money-input"). An A3 agent reading its own ownership list builds a duplicate. **Fix:** strike `money-input` from A3's owns list (lines 50, 52); line 83 is the single source of truth.
- **P2-04 · Phase 2 · Task 4 buildAusgabenWhere/buildEinnahmenWhere; Task 1 listKategorieOptions** — the WHERE builder matches `inArray(kategorieNameSnapshot, …)` but Task 1 never pins whether `listKategorieOptions(kind)` returns the name-snapshot or the id. If Phase 3 returns ids (the natural option-loader shape), every kategorie filter silently returns zero rows — no typecheck error. **Fix:** state in Task 1 (and Task 4) that `listKategorieOptions` MUST return `value = kategorieNameSnapshot` string, not the id.
- **P4-01 · Phase 4 · amendment block vs Task 4/5 bodies** — the amendment correctly routes Mitglied/Extern + Schon-bezahlt through `markExpenseErstattet` (mails), but Task 4 Step 3 still calls `markExpenseAsPaid({…, notify})` — wrong helper, positional-vs-object signature, and a non-existent `notify` key. - **P1-01 · Phase 1 · Task 8 step 3 vs transform.ts:691-717** — the plan asserts importer donations "are geldspende so the Sachspende CHECK won't bite," but transform.ts:692 emits `spendeKind='sachspende'` when the legacy column contains "sach"; once Task 10's CHECK lands, those inserts violate it and the importer ships broken. **Fix:** in transform.ts force `spendeKind='geldspende'` on the sheet-import donation path (legacy sheets carry no Wertermittlung data); remove the false "these are geldspende" assertion.
- **UX-01 · Phase 3 Task 7/10 + Phases 4/5/6 · TransactionListScaffold** — the daily-driver action "create new booking" has no entry point: the scaffold's locked prop interface has no slot for it, so three parallel C-tracks each invent their own button. **Fix:** add `newHref: string` + `newLabel: string` props to TransactionListScaffold in Phase 3 Task 7; scaffold renders one primary CTA (desktop top-right, mobile ≥44px sticky/FAB); each C-track passes its German label + href.
